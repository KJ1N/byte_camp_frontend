# 文舟 AI Creator Hub 效果评估报告

## 1. 评估范围

本报告评估当前 MVP 的五类交付指标：

- AI 生成：文章初稿、标题优化、正文改写、合规改写和多模态图文生成。
- 内容审核：发布前文字审核、图片审核聚合、WARN/BLOCK 处理和审核记录。
- 分发效果：推荐信息流、热点榜、爆文榜、互动事件和创作者数据反馈。
- 性能表现：首页、榜单、详情页的首屏与滚动加载能力。
- 工程化：模块边界、类型检查、单元测试、接口层测试、E2E 和 CI。

评估依据来自当前代码实现、种子数据、自动化测试、Playwright E2E 脚本和本地验证命令。

## 2. AI 生成评估

### 2.1 功能覆盖

当前 AI 创作链路已覆盖：

- `POST /ai/generate-article/stream`：流式生成标题、大纲和正文。
- 标题优化：生成多个标题候选，并在前端逐步呈现。
- 正文改写：支持润色、扩写、缩写和风格转换。
- `POST /audit/rewrite/stream`：对 WARN/BLOCK 内容生成合规替代表达。
- `POST /ai/generate-multimodal/stream`：生成正文、图片计划、图片状态和完成图片 URL。

后端通过 AI Gateway 统一处理模型配置、Prompt 装配、结构化输出、SSE 事件、超时、重试和 mock 降级，符合“AI 能力集中在后端”的架构边界。

### 2.2 评估结论

| 指标 | 当前结论 |
| --- | --- |
| 生成链路完整性 | 已完成从主题输入到草稿保存的主链路 |
| 流式体验 | 文章生成、标题优化、正文改写、合规改写均有 SSE 支持 |
| 降级能力 | 支持 `auto` / `mock` / `live` 三种模式，离线演示可使用 mock |
| 多模态能力 | 已有独立多模态工作台，生成图文可继续进入草稿、审核和发布链路 |
| 可观测性 | AI 请求会记录 requestId、模型、耗时、token 和状态到结构化服务日志 |

主要风险：

- 真实模型输出仍可能出现 JSON 不稳定，当前通过 parser 和错误处理兜底。
- 文生图耗时较长，后续适合接入 Redis + BullMQ 做异步任务队列。
- 生成图片 URL 可能存在过期风险，长期生产环境建议转存到稳定云存储。

## 3. 审核与质量评分评估

### 3.1 功能覆盖

当前发布前处理链路为：

```text
草稿 -> 审核 -> 质量评分 -> 发布快照 -> 文章详情 -> 分发
```

已完成能力：

- `POST /audit/check`：按 draftId 执行发布前审核。
- `POST /scoring/article`：生成五维质量分和综合分。
- `POST /publish/:draftId`：审核通过后发布文章快照。
- WARN 内容返回风险原因和修改建议，引导用户修改后重审。
- BLOCK 内容禁止发布，不生成文章。
- 草稿中图片节点会被拆分出来进行图片审核，文字和图片结果聚合为最终裁决。
- 审核记录和质量分会持久化，文章详情页可展示发布审核和评分信息。

### 3.2 评估结论

| 指标 | 当前结论 |
| --- | --- |
| 发布前强审核 | 已实现，发布动作由后端统一拦截 |
| 高危内容拦截 | BLOCK 不可发布 |
| 中风险改写 | WARN/BLOCK 可触发合规改写流 |
| 可解释性 | 返回风险片段、类别、建议和审核摘要 |
| 质量评分 | 覆盖内容价值、表达质量、读者体验、传播潜力、合规安全五个维度 |
| 图文审核 | 支持正文与图片分开审核后聚合裁决 |

主要风险：

- 审核和评分效果依赖模型能力，仍需要用更大样本集持续评估误杀率和漏检率。
- 当前没有独立运营后台复核页，审核记录主要服务发布页和详情页展示。

## 4. 分发与数据反馈评估

### 4.1 功能覆盖

当前分发链路包含：

- `GET /feed`：推荐信息流，首页从后端读取内容。
- `GET /rankings/hot`：热点榜。
- `GET /rankings/top`：爆文榜。
- `GET /articles/:id`：文章详情。
- `POST /articles/:id/events`：阅读、点赞、收藏等互动事件。
- Creator 页面展示真实作品管理、阅读量、互动数据、平均质量分和统一“我的内容”列表。

榜单使用 Redis Sorted Set 缓存，Redis 不可用时回退 PostgreSQL 查询与内存排序。互动数据会影响榜单排序和创作者数据反馈。

### 4.2 评估结论

| 指标 | 当前结论 |
| --- | --- |
| 信息流 | 首页已接入后端推荐流 |
| 热点榜/爆文榜 | 已按热度、质量、时间衰减和互动数据排序 |
| 无限滚动 | 首页、榜单支持分页或滚动加载 |
| 互动反馈 | 阅读、点赞、收藏事件已写入并进入展示/排序 |
| 创作者反馈 | Creator 页面接入真实统计和内容状态 |
| 选题素材 | 每日 AI 资讯和热点资讯可预填工作台 |

主要风险：

- 当前排序是可解释的规则模型，不是个性化推荐。
- 榜单质量依赖种子数据和真实互动规模，线上早期需要补充冷启动策略。

## 5. 性能评估

### 5.1 当前性能措施

- Next.js App Router 页面按路由拆分，详情页和榜单动态渲染。
- 首页、榜单和信息流使用分页或无限滚动，避免一次加载全部内容。
- 榜单使用 Redis 缓存排序结果，Redis 失败时回退 PostgreSQL。
- 每日资讯使用 Redis 当天快照和最近非空快照，减少外部接口调用。
- AI 生成使用 SSE 分段返回，减少用户等待空白时间。
- `scripts/e2e-rankings-performance.mjs` 提供榜单 LCP 与无限滚动性能 E2E。

### 5.2 评估结论

| 指标 | 当前结论 |
| --- | --- |
| 榜单首屏 LCP | 已配置 Playwright 性能 E2E，阈值默认 2500ms |
| 榜单滚动加载 | 性能 E2E 覆盖热点榜和爆文榜滚动加载 |
| 首页性能 | 已采用分页加载，仍需线上 Lighthouse 补充正式报告 |
| AI 响应体验 | SSE 逐步渲染，避免长时间无反馈 |
| 外部资讯性能 | Redis 快照减少重复请求，外部失败不阻塞主创作链路 |

后续建议：

- 部署后使用 Lighthouse 对首页、榜单页、文章详情页分别采集 LCP、CLS、INP。
- 根据真实部署环境补充 CDN、图片缓存和 API 压缩配置。

## 6. 工程化评估

### 6.1 模块边界

| 模块 | 当前职责 |
| --- | --- |
| `apps/web` | UI、路由、编辑器、前端状态、SSE 消费和本地暂存 |
| `apps/api` | 鉴权、AI 编排、审核、评分、发布、榜单、素材和数据持久化 |
| `packages/shared` | DTO、枚举、共享类型、评分和内容结构契约 |

当前实现没有把 AI key、审核裁决、评分或榜单排序迁移到前端，符合后端集中编排要求。

### 6.2 测试与 CI

当前自动化覆盖：

- `pnpm typecheck`：前后端与 shared 类型检查。
- `pnpm test`：Service、Controller、helper 和共享链路测试。
- `pnpm build`：shared、API 和 Web 构建。
- `pnpm test:e2e:smoke`：登录 -> 创作 -> 保存草稿 -> 审核评分 -> 发布 -> 查看详情。
- `pnpm test:e2e:rankings-performance`：榜单 LCP 和无限滚动。

本轮已补齐接口层测试：

- Auth：注册、登录 Controller 转发。
- Drafts：创建、列表、详情、更新、删除、版本列表、版本回滚。
- Audit：审核检查和合规改写 SSE。
- Scoring：质量评分接口转发。
- Publish：发布、公开详情、撤回。

### 6.3 评估结论

| 指标 | 当前结论 |
| --- | --- |
| 类型安全 | 已覆盖全 workspace typecheck |
| 单元测试 | 主要业务服务和前端 helper 已覆盖 |
| 接口层测试 | Auth、Drafts、Audit、Scoring、Publish 已补齐 Controller 层测试 |
| E2E | 已覆盖主链路 smoke 和榜单性能 |
| CI | 已包含 install、Prisma generate、typecheck、test、build |
| 文档 | PRD、架构、审核规则、评估方案、dev plan 和本报告已齐备 |

## 7. 总体结论

当前项目已经达到训练营 MVP 的核心交付要求：

- 用户可以完成登录、创作、编辑、审核、评分、发布、浏览和互动反馈闭环。
- AI 能力不止停留在单点生成，已经覆盖写作、改写、审核改写、多模态和每日选题素材。
- 发布链路由后端强制执行审核和评分，具备可解释记录。
- 内容可以进入信息流、热点榜和爆文榜，并受互动数据影响。
- 工程结构清晰，具备类型检查、单元测试、接口层测试、E2E 和 CI。

上线前建议优先完成：

1. Web、API、PostgreSQL、Redis 的生产部署和环境变量校验。
2. 线上 Lighthouse 性能采集，补充首页、榜单、详情页真实指标。
3. 真实模型和真实资讯接口的稳定性压测。
4. 更大规模审核样本评估，统计 BLOCK/WARN/PASS 的误杀率和漏检率。
