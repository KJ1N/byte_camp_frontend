# 项目协作与开发规则

## 文档优先

本项目所有开发都必须以 `docs/` 目录下的文档为准。

开发、设计或修改代码前，需要先阅读：

- `docs/PRD.md`
- `docs/architecture.md`
- `docs/audit-rules.md`
- `docs/evaluation-plan.md`

如果代码需求与现有文档冲突，需要先更新相关文档，或向用户确认产品目标、架构边界和实现方向后再继续。

## 关键开发点先写技术方案

每到一个关键开发点，正式写代码前，必须先在以下目录生成该部分的技术文档：

```text
docs/dev_plan/
```

技术文档至少需要包含：

- 功能目标和用户价值
- 涉及的前端、后端、共享包模块
- 需要新增或修改的文件
- 数据模型或 API 变更
- 前端交互与页面状态设计
- 后端服务、鉴权、审核、评分、发布或榜单逻辑
- 风险、边界情况和回滚方案
- 验证方式和测试计划

技术文档生成后，必须先请用户确认。用户确认无误后，才能按照总架构和该技术文档开始开发。

## 实现规则

开发时必须同时遵守：

1. `docs/architecture.md` 中定义的整体技术架构
2. `docs/dev_plan/` 中已确认的对应技术方案

模块边界保持清晰：

- `apps/web` 只负责 UI、路由、编辑器交互和前端状态。
- `apps/api` 负责业务逻辑、AI 编排、审核、评分、发布、榜单和数据持久化。
- `packages/shared` 负责前后端共享类型、枚举、常量和 API 契约。

不得把 AI 密钥、审核裁决、质量评分、榜单排序等后端职责迁移到前端。

页面主体样式实现时必须使用 Tailwind CSS utility class 搭建页面主体样式，减少新增全局 CSS。

## 工作台页面风格规范

创作工作台和草稿编辑页以用户提供的今日头条发布页参考图为视觉模板：

- 使用 Tailwind CSS utility class 搭建页面主体样式，减少新增全局 CSS。
- 整体为浅灰应用背景，顶部白色导航栏，主体采用左右双栏：左侧为白色编辑画布，右侧为浅色 AI 创作助手面板。
- 页面气质应贴近生产工具：安静、克制、高密度、留白充足，不做营销式 hero、不使用装饰性渐变背景或大卡片堆叠。
- 编辑区保持类似内容发布后台的结构：顶部工具栏、标题输入、正文编辑区、底部固定或粘性操作栏。
- 右侧助手面板保持独立栏位，包含“AI 创作 / 内容建议”标签、生成内容预览、添加正文/复制/重试按钮和底部输入框。
- 主色使用接近头条发布页的红色强调，辅助按钮可使用轻微红到紫的渐变；其余区域以白、浅灰、深灰文字和细边框为主。
- 圆角保持克制，常规容器和按钮优先使用 6px 到 8px；不要使用大圆角胶囊风格。
- 字号层级偏工具化：顶部与工具栏 14px 左右，正文编辑 16px 到 18px，标题输入 28px 到 32px；避免 hero 级大标题出现在工作台内。
- 图标按钮优先使用简洁符号或已有图标库，按钮需有明确 hover/focus 状态和可访问标签。
- 移动端改为单列布局：编辑区在前，AI 助手面板下移或可折叠，底部操作区不能遮挡正文输入。

## GitHub 提交规则

每次修改完成后，都必须询问用户是否需要提交并推送到 GitHub。

提交前必须向用户确认提交说明内容。

未获得用户明确确认前，不得自动提交或推送。

用户需要同时确认：

- 是否将本次修改提交并推送到 GitHub
- 本次提交使用什么提交说明

如果工作区存在与当前任务无关的改动，需要明确指出，并且不得擅自把无关改动加入提交。

提交说明默认使用中文。

## 验证规则

提交前应运行与修改内容相关的检查：

- `pnpm typecheck`
- 修改运行入口时运行对应 build 命令
- 有测试覆盖时运行相关测试
- 修改 Prisma schema 时运行迁移或 schema 校验

如果某项检查无法运行，需要说明原因。
